## 一、什么是 DimOS（What）

**一句话定义：DimOS 是一个面向"通用机器人"的智能体操作系统**——它给你一套搭积木的零件，
让你把"相机感知 / 路径规划 / 电机控制 / 自然语言对话"这些原本互不相干的子系统，
用统一的方式拼起来，运行在一台真实或仿真的机器人上。

它不是一个 Python 库，也不只是一个框架。它更像是一个**约定的工厂车间**：你把零件按约定的接口
做好，车间会负责把它们装在不同的工人手里（不同的操作系统进程），把传送带（数据流）接起来，
最后由一个总调度（蓝图）把整条流水线开起来。

如果一定要从代码角度概括，DimOS 的"零件清单"只有 5 种东西：

| 中文术语 | 英文名 | 类比 |
| --- | --- | --- |
| 模块 | `Module` | 厨房里的电器 |
| 蓝图 | `Blueprint` | 厨房的菜谱+接线图 |
| 技能 | `Skill` | 给机器人贴的"动作技能卡" |
| 传输 | `Transport` | 电器之间的电线 / 管道 |
| 全局配置 | `GlobalConfig` | 整个家的总闸 |

下面我们用日常生活里的厨房做类比，把这 5 个概念过一遍。

### 1. 模块（Module）≈ 厨房里的电器

把一台机器人想象成一间智能厨房。你的厨房里有冰箱、烤箱、洗碗机、抽油烟机……
每一台电器**只做一件事**：

- 冰箱：保存食材。
- 烤箱：把食物加热。
- 洗碗机：洗碗。

它们彼此**独立运行**——洗碗机坏了，冰箱照常制冷；烤箱里在烤东西，不影响你打开冰箱。
你不需要知道烤箱内部的电路图，你只需要知道**它输入什么、输出什么**：

> 烤箱：输入一盘生肉 + 一个温度 → 输出一盘熟肉。

DimOS 中的 `Module`（模块）就是这种"电器"。一个模块代表机器人身上一个**自主运行的子系统**，
比如：

- 相机模块：输入电源 → 输出一串图像（像冰箱"输出冷气"）。
- 物体检测模块：输入图像 → 输出"看到了什么物体、在哪儿"。
- 导航模块：输入"目标点 + 当前地图" → 输出"机器人下一步该往哪走"。

每个模块对外只暴露两类接口：

1. **数据流接口**（`In[T]` / `Out[T]`）：用类型 `T` 声明"我输入什么、输出什么"。例如相机模块的
   `color_image: Out[Image]` 表示"我对外输出一条名字叫 `color_image`、类型是 `Image` 的数据流"。
2. **生命周期方法**：通常是 `start()` 和 `stop()`，相当于电器的开关。

后续章节会展开讲数据流的类型系统、生命周期、异步处理等细节。在本章你只需要记住一句话：

> **模块 = 输入若干条数据流、输出若干条数据流的"黑盒电器"。**

### 2. 蓝图（Blueprint）≈ 菜谱里的接线图

一台机器人只装一个相机模块、什么都不连，能做的事情非常有限。
真正有用的"机器人能力"都是若干个模块**协同工作**的结果，例如：

> 「跟随一个人」 = 相机 → 检测人 → 估计人的位置 → 算出我该往哪走 → 发电机指令。

这条流水线在 DimOS 里叫 **`Blueprint`（蓝图）**。蓝图就是一张接线图，告诉系统：

- 要用到哪些模块？
- 哪个模块的输出口，要接到哪个模块的输入口？
- 这些模块整体启动起来，叫什么名字？

继续厨房类比：蓝图就像一张写好的**家常菜菜谱**，菜谱告诉你：

> 「先开冰箱拿出蔬菜，再开洗菜池洗，再交给案板切，再交给炒锅炒。」

DimOS 让你用 `autoconnect(...)` 把一组模块拼接成一张蓝图。它最聪明的地方在于
**自动按 `(数据流名字, 数据类型)` 匹配**，
你不需要手动写"模块 A 的 image 接到模块 B 的 image"——系统会自动发现匹配。

> **蓝图 = 一份「装哪些模块 + 它们之间怎么连」的菜谱。**

### 3. 技能（Skill）≈ 机器人的"动作技能卡"

让我们再升级一下厨房的设定：现在你想让一个**会说话的助手**站在厨房里，
它能听你说"帮我热一下牛奶"，自己跑去开微波炉、按 30 秒、按启动。
这位助手要能做到这件事，前提是它**手里有一沓技能卡**：

- 「打开微波炉」卡
- 「设置加热时间」卡
- 「按启动键」卡

每张卡上都写着：这个动作能做什么、需要哪些参数、做完会返回什么。

DimOS 里的 `Skill`（技能）就是这种**贴在机器人身上、专门给 AI 用的动作卡**。
在代码上，技能是模块里被特殊装饰器 `@skill` 标记过的一个方法，例如机器人狗 Unitree Go2
就有 `move(x, duration)`（前进或后退）、`speak(text)`（发声）这一类技能。

技能的特殊之处在于：

- 它会被自动收集，**暴露给大语言模型（LLM）作为可调用的工具**。
- 它有一组严格的规则：必须写文档字符串、必须有类型标注、必须返回字符串。
  这些规则的细节是 `02-agents.md` 的核心内容，本章不展开。

> **技能 = 一个被特别标注、能被 AI 直接调用、最终落到电机/喇叭上的动作。**

### 4. 传输（Transport）≈ 电器之间的电线 / 管道

回到厨房：冰箱和烤箱之间没有直接相连，你要把食物从冰箱"传"到烤箱，
靠的是**你自己的手**或者一条**传送带**。同样的，模块之间也要靠某种"线"才能传消息。

DimOS 把这种"线"抽象成 `Transport`（传输）。同一条数据流可以根据**传输介质不同**，
选择不同的"线材"。AGENTS.md 列出了 5 种常用传输：

- **`LCMTransport`**（默认）：基于多播 UDP，是大多数小消息（控制指令、状态量）的默认通道。
  LCM 是一种轻量级机器人通信协议，你可以理解成"一根普通的电线，所有人都能监听"。
- **`SHMTransport` / `pSHMTransport`**：基于共享内存，**专门给图像、点云这种"大而频繁"的数据**用。
  共享内存意味着两个进程指向同一段物理内存，避免数据复制。
- **`pLCMTransport`**：在 LCM 上跑 pickle 序列化，用来传比较复杂的 Python 对象。
- **`ROSTransport`**：把 DimOS 的数据流和 ROS 的话题（topic）打通，方便和已有的 ROS 节点互通。
  ROS 是一个老牌的机器人中间件，本章只需要知道"它存在、DimOS 可以跟它说话"即可。
- **`DDSTransport`**：使用 DDS 协议进行发布/订阅，需要安装额外的依赖。DDS 是另一种业界主流的
  机器人 / 工业通信协议。

不同的传输有不同的取舍：要不要复制内存？能不能跨机器？能不能给 ROS 看？
你**通常不需要关心**——蓝图里默认会用 `LCMTransport`，你只有在做图像流的时候才需要
显式切换到 `SHMTransport`。

> **传输 = 模块之间真正"那根线"。普通消息走默认线，图像点云走共享内存线，
> 需要跟 ROS 互通就走 ROS 桥。**

### 5. 全局配置（GlobalConfig）≈ 一家人的总闸

最后是 `GlobalConfig`，全局配置。它管的是**整套机器人启动时要回答的一些"全局问题"**：

- 真实机器人的 IP 是多少？（`robot_ip`）
- 是连真机器人，还是用仿真（MuJoCo）？（`simulation`）
- 还是干脆只用录好的数据回放？（`replay`）
- 用什么可视化界面？（`viewer`，例如 `rerun` 或 `foxglove`）
- 启动多少个工人进程？（`n_workers`）
- 给大模型暴露 skill 的 HTTP 端口是几号？（`mcp_port`，默认 9990）

这些都是单台机器人**整体的设定**，不属于任何一个模块。继续类比：你家所有电器
都要看一下"今天家里跳闸了吗？"（总闸），都要看一下"现在是 220V 还是 110V？"。
全局配置就扮演这个"总闸 + 电压"的角色。

`GlobalConfig` 的取值有一个**层级覆盖关系**（cascade）：

```
默认值 → .env 文件 → 环境变量 → 蓝图里写死的值 → 命令行 --xxx 参数
                                                       (优先级最高)
```

也就是说，你可以在 `.env` 里写一份默认配置，再在命令行用 `--robot-ip 192.168.123.161`
临时改一次。具体每个字段是什么含义、覆盖顺序怎么实现，留到 `01-core.md` 详细讲。

## 二、为什么需要它（Why）

读到这里，你也许会想："这么麻烦，我直接写一个 Python 大脚本，里面相机、检测、控制、对话
全部塞在一起，不也能跑吗？"——这是几乎所有初学者第一次写机器人代码会做的事，也是
**几乎所有人很快就会撞到墙**的方案。下面三段我们对比一下三种写法。

### 写法一：一个超大 Python 脚本——会怎样失败？

假设你只用一个文件 `robot.py`，里面写了 2000 行代码，开头读相机帧、中间跑神经网络做物体检测，
后面用 ROS-like 的方式给机器人发速度指令，最后再启一个聊天机器人接收语音。
新手写到大概第 500 行的时候，会陆续遇到这些问题：

1. **Python 的 GIL 让你用不上多核 CPU**。神经网络跑在主线程里，会把整个程序卡住，
   导致控制循环延迟很大。
2. **任何一个子系统崩了，整个机器人都跟着挂**。检测模型加载失败 → Python 进程退出 → 控制
   指令也发不出去 → 机器人在原地呆住或者更糟糕地撞墙。
3. **代码越长越乱**。相机 / 检测 / 控制 / 对话四块互相调用彼此的函数，没几天你就忘了
   "这个变量到底是哪个模块改的"。
4. **复用接近为零**。你换一台机器人（比如从机器狗换到人形机器人）只是控制层改了，
   但你的脚本里"控制"和"感知"是粘成一团的——不重写就跑不起来。

简单来说，一个大脚本对**单人小项目**勉强够用，但只要任务复杂一点、要在真实硬件上长时间跑，
它就会以非常痛苦的方式崩溃。

### 写法二：直接写 ROS——又会怎样？

ROS（Robot Operating System）是过去十几年最流行的机器人中间件，它解决了"多进程 + 类型化消息流"的
基础问题。如果你直接用 ROS，确实可以把上面那个 2000 行脚本拆成几个独立的 node（节点）。
但 ROS 又带来了一组新的麻烦：

1. **类型系统在 Python 端非常薄**。ROS 消息基本是 dict，写错字段名要等到运行时才报错。
2. **节点之间的"接线"写得很啰嗦**。你要写一大堆 `rospy.Publisher / Subscriber("/foo", Msg, ...)`
   的样板代码，并且话题名是字符串，重命名一个话题需要全局搜索改字符串。
3. **它不关心"AI 智能体"这一层**。ROS 本身没有"把一个方法暴露给大语言模型当工具"这种概念。
   如果你要在 ROS 上接 LLM，要自己造一整套桥。
4. **多机器人 / 多形态适配仍然很难**。ROS 的"启动 launch 文件"语法很灵活但很容易写错，
   你要用宏来组合不同的机器人，很快就会爬出脚本式 launch 的复杂度。

简单说：ROS 解决了"多进程数据流"的事，但**没解决"组合多个机器人形态 + 接 LLM 智能体"
这种现代任务**。

### 写法三：DimOS 的模块 + 蓝图——改善在哪？

DimOS 不是想替代 ROS（它甚至可以通过 `ROSTransport` 跟 ROS 互通），它解决的是另一层问题：
**怎么把感知 / 规划 / 控制 / 语言这四种系统拼起来，写得既快又不容易爆**。

它的核心改进有三点：

1. **类型化数据流 + 自动接线**。你声明 `color_image: In[Image]`，类型系统会在蓝图组装时
   自动把同名 + 同类型的输入输出接起来；改错字段名编辑器立刻报错，不用等运行时。
2. **进程隔离（forkserver workers）**。每个模块跑在自己的 fork-server 工人进程里，
   一个模块崩溃**不会带垮整台机器人**；CPU 利用率也不再受 GIL 限制。
3. **蓝图可组合**。一个机器人的全套能力是若干小蓝图叠加而成的，你可以单独拿出"导航 + 感知"
   的子蓝图，在另一个机器人上复用。整体仍然是普通的 Python 代码，没有额外的 launch DSL。

加上 `Skill` 抽象，DimOS 还能把这些动作能力一键暴露给大语言模型，这是 ROS 时代没有的。

> 一句话概括："**泛用机器人需要把感知 / 规划 / 控制 / 语言这四种系统拼起来，
> 靠类型化的数据流和进程隔离才能不爆炸**"——这就是 DimOS 存在的理由。

## 三、如何使用（How）

本节给你一个**最小可读的、不需要硬件**的心智模型。我们不演示真正跑起来一台机器人——那是
后续章节的事。本节只回答两个问题：

1. 一段 DimOS 程序的**生命周期**长什么样？
2. 启动一台真实/回放机器人的**命令行**长什么样？

### 3.1 最小生命周期：四个步骤

DimOS 程序的**典型骨架**永远是同一条流水线：

```
1. 定义 Module        # 写好你的相机模块、检测模块……
2. autoconnect 装配   # 把若干模块拼成一张 Blueprint
3. build()            # 让 DimOS 把模块部署到 forkserver 工人进程里、把传输接起来
4. loop()             # 阻塞主线程，等待 Ctrl+C；按下后自动停止所有模块
```

这四步可以用一行简单的代码串起来（来自 `AGENTS.md` 的 Blueprints 段落）：

```python
autoconnect(module_a(), module_b(), module_c()).build().loop()
```

第一次看到这一行不需要看懂细节。你只需要记住四个动词：**定义、装配、构建、运行**。

### 3.2 最小代码骨架：一个 `MyModule`

下面这段代码完全来自 `AGENTS.md` 的 "Modules" 段落，只做了非常轻微的简化（把示例函数体
换成一句注释）。它演示的是"一个模块从 `In[Image]` 读图，加工后从 `Out[Image]` 发出去"：

```python
from dimos.core.module import Module
from dimos.core.stream import In, Out
from dimos.core.core import rpc
from dimos.msgs.sensor_msgs import Image


class MyModule(Module):
    color_image: In[Image]      # 输入：一条名为 color_image 的图像流
    processed: Out[Image]        # 输出：一条名为 processed 的图像流

    @rpc
    def start(self) -> None:
        super().start()
        # 订阅输入流：每来一帧，就调用 self._process
        self.color_image.subscribe(self._process)

    def _process(self, img: Image) -> None:
        # 这里写你真正的图像处理逻辑：滤波、裁剪、检测……
        # 处理完之后用 .publish(...) 把结果发到输出流上。
        self.processed.publish(img)  # 示例：原样转发
```

逐行解释（你只需要看一眼，不必背下来）：

- `class MyModule(Module)`：继承 `Module` 基类，这一行让你拥有"模块"的所有特性。
- `color_image: In[Image]`：声明一条输入数据流，名字叫 `color_image`，类型是 `Image`。
- `processed: Out[Image]`：声明一条输出数据流，名字叫 `processed`。
- `@rpc` 装饰器：表示这个方法可以被 DimOS 的进程间机制调用，**生命周期方法**（`start` / `stop`）
  都需要这个装饰器。本章先把它当成"系统要求的固定写法"，01 章会展开。
- `subscribe(self._process)`：告诉 DimOS"当 `color_image` 流有新数据时，调用 `_process`"。
- `publish(img)`：把数据塞进输出流，下游模块就能收到。

把这个 `MyModule` 与另一个会发图像的模块（比如 `CameraModule`）一起放进 `autoconnect()`，
DimOS 会发现 `CameraModule.color_image: Out[Image]` 和 `MyModule.color_image: In[Image]`
名字与类型都对得上，**自动接线**——你不必写任何"connect"代码。

### 3.3 最小 CLI：跑一台 Unitree Go2 机器人狗

代码骨架不实用——大多数时候你不会自己手写一台机器人的所有模块。DimOS 已经为常见硬件
（机器狗、人形、机械臂、无人机）写好了一组**现成的蓝图**，你只需要在命令行里挑一个用。
最简单的两条命令是（直接抄自 `AGENTS.md` Quick Start 段落）：

```bash
dimos --replay run unitree-go2
```

把这条命令逐段拆开看，每一段在做什么：

- **`dimos`**：DimOS 的命令行入口程序。你装好包以后就直接能用，相当于 `git` 或 `docker`。
- **`--replay`**：一个**全局开关**（`GlobalConfig` 字段 `replay`）。打开它表示**不连真机器人**，
  而是从录好的数据集回放传感器数据。新手强烈建议先用这个模式，硬件没准备好也能跑。
- **`run`**：子命令，"运行一个蓝图"。
- **`unitree-go2`**：蓝图名字，意思是"Unitree 公司的 Go2 机器人狗"。这个蓝图里通常包含
  相机、感知、建图等基础模块，但**不包含 LLM 智能体**。

执行后，DimOS 会按 3.1 的生命周期把这个蓝图里的所有模块部署到 forkserver 工人进程，
把数据流接起来，开一个可视化窗口（默认是 Rerun），然后阻塞等待你按 Ctrl+C。

### 3.4 给同一台机器人接上 LLM 智能体

把上面这条命令稍微改两个地方，就能从"会感知的机器人"升级成"会听话的机器人"：

```bash
dimos --replay run unitree-go2-agentic --daemon
```

差别有两点：

- **蓝图换成了 `unitree-go2-agentic`**。这个蓝图在原来的基础上**多加了两个模块**——
  - 一个 `McpServer`（MCP 服务器）模块，在本地起一个 HTTP 服务（默认端口
    `9990`，对应 `GlobalConfig.mcp_port`），把这台机器人的所有 `@skill` 方法以
    HTTP 工具的形式暴露出去。
  - 一个 `McpClient`（MCP 客户端）模块，它就是 LLM 那一侧的智能体：从 `McpServer`
    拉到工具列表，收到你发的文本指令后，让 LLM 决定调用哪些工具。
  - 这两个模块**必须两个都加**才构成一个完整的 agentic 蓝图——`McpServer` 负责"把
    技能挂到墙上"，`McpClient` 负责"读墙上的菜单点菜"，缺一个就跑不起来。本系列
    `02-agents.md` 会详细讲 MCP。
- **加了 `--daemon`**：让程序在后台运行，不占着当前终端。

跑起来之后你可以用：

```bash
dimos agent-send "walk forward 2 meters then wave"
```

向后台运行的智能体发一句话。智能体会自己把这句话翻译成"先 move 再 wave"两次技能调用。
具体到怎么写技能、怎么调技能，留到 `02-agents.md` 详细讲。

这一节的关键点不是让你会用所有 CLI，而是让你**把"代码骨架"和"实际能跑的机器人"在脑子里
连成一条线**：你写的就是 `MyModule` 那样的模块，机器人厂商或者你自己把若干这样的模块
拼成一张蓝图，最后 `dimos run` 把蓝图跑起来——仅此而已。

## 四、与其他模块的关系

DimOS 的整体结构可以画成下面这张简化关系图。它有意把细节抹掉，只保留 5 个核心方块：

```
                       ┌─────────────────────┐
                       │    GlobalConfig     │
                       │   （全局总闸）       │
                       └─────────┬───────────┘
                                 │ 注入
                                 ▼
       ┌───────────┐       ┌───────────┐       ┌───────────┐
       │  Module A │──────▶│  Module B │──────▶│  Module C │
       │ (相机)    │ Trans │ (检测)    │ Trans │ (控制)    │
       └───────────┘ -port └───────────┘ -port └───────────┘
             ▲                                       │
             │              ┌──────────────────────┐ │
             └──────────────│      Blueprint       │◀┘
                            │ (用 autoconnect 装配) │
                            └──────────┬───────────┘
                                       │ build() + loop()
                                       ▼
                                ┌──────────────┐
                                │  dimos run   │
                                │  （CLI 入口） │
                                └──────────────┘
                                       ▲
                                       │ 调用
                              ┌────────┴────────┐
                              │ Skill (@skill)   │
                              │ ← 给 LLM 用的工具 │
                              └─────────────────┘
```

读这张图的方法：

- 三个 `Module` 方块用 `Transport`（电线）连起来，箭头表示数据流方向。
- 蓝图 (`Blueprint`) 把这三个模块"包"在一起。
- `GlobalConfig` 像总闸，挂在最上面，每个模块都能读到全局配置。
- `dimos run` 是命令行入口，它根据你给的蓝图名字，去执行 `build() + loop()` 那条流水线。
- `Skill` 是可选的一层：当蓝图里同时包含 `McpServer` 和 `McpClient` 时，模块上的
  `@skill` 方法会被自动暴露成 LLM 可调用的工具（`McpServer` 在 HTTP 上挂出工具，
  `McpClient` 是 LLM 那侧把工具调起来的客户端）。

### 一句话总结

> **DimOS = 模块 + 蓝图 + 技能 + 传输 + 全局配置。**
>
> 你写**模块**，描述"输入什么、输出什么、做什么"。你用 **`autoconnect`** 把模块拼成**蓝图**。
> 模块之间通过 **`Transport`** 这种"线"传消息，整个系统读一份共享的 **`GlobalConfig`**。
> 当你把蓝图里再加上 `McpServer` + `McpClient` 这一对 MCP 桥，模块身上的 **`@skill`**
> 就会被暴露给 LLM（`McpServer` 把技能挂成 HTTP 工具，`McpClient` 是 LLM 那侧的智能体），
> 机器人就开始"会说话"。这一切由 `dimos run <蓝图名>` 一行命令启动。

到这里你应该已经能在心里画出 DimOS 的整体形状了。下一章 `./01-core.md` 我们就
钻进 `dimos/core/` 这个仓库里最核心的目录，去看每一个零件具体是怎么实现的。
