# DimOS 未来可优化方向（路线图草案）

本文汇总**面向未来的优化主题**，供维护者与贡献者讨论与排期之用；所列项不代表承诺或优先级排序。部分内容对照了当前仓库中的已知约束（例如依赖、测试分层、CI 行为）。

---

## 1. 依赖与安装面（dependencies / install footprint）

| 方向 | 为什么重要 | 大致思路 | 技能领域 |
|------|------------|----------|----------|
| 弱化「全家桶」式硬依赖 | 体积、安装时间、在无显示/无 GPU 环境下的可用性都会受影响；`pyproject.toml` 中已有注释希望将部分组件变为可选。 | 梳理 DimOS 核心与子系统边界，把可视化、特定机器人 SDK、重型仿真等逐步挪到 optional extra 或 lazy import；在文档中明确「最小安装」与「全量安装」路径。 | 打包、架构 |
| 可选可视化栈（rerun 等） | 同上；与核心消息流、调试回本不应强绑定。 | 完成「无 rerun 亦可运行」的路径，并把 viewer 相关调用收口到可选模块。 | 核心、可视化 |

---

## 2. 模块、进程与协调（modules / workers / coordination）

| 方向 | 为什么重要 | 大致思路 | 技能领域 |
|------|------------|----------|----------|
| Worker 启动与生命周期 | 多进程 forkserver 模型在冷启动、内存占用、调试体验上有优化空间。 | 度量不同蓝图下的进程数与启动时延；探索延迟加载、共享只读状态、更清晰的 shutdown 语义。 | 核心、运行时 |
| RPC 与跨模块调用 | Spec 注入已在构建期保证类型安全；运行时仍可能有序列化、超时与失败传播方面的痛点。 | 统一超时/取消策略、可观测的 RPC 指标、错误信息对 Skill/Agent 更友好。 | 核心、Agent |

---

## 3. 传输与数据平面（transports / data plane）

| 方向 | 为什么重要 | 大致思路 | 技能领域 |
|------|------------|----------|----------|
| 高带宽话题（图像、点云） | LCM 默认易用，大图/点云更适合共享内存等路径（见仓库中与 transport 相关的用法说明）。 | 审计热点话题的默认 transport；为常见蓝图提供经评测的推荐配置。 | 传输、感知 |
| 背压与流控 | Reactive 管线在复杂图上容易出现积压或丢帧策略不统一的问题。 | 在 `docs/usage` 中数据流/传感器流相关约定基础上，收敛默认策略与可调参数。 | 流式处理 |

---

## 4. Agent、MCP 与 Skills（agent stack）

| 方向 | 为什么重要 | 大致思路 | 技能领域 |
|------|------------|----------|----------|
| Skill schema 与提示词维护成本 | 文档与 schema 规则较严（docstring、注解、返回 `str` 等），扩展 Skill 时易踩坑。 | 生成器/静态检查辅助、示例模版、与 system prompt 的同步检查。 | Agent、工具链 |
| MCP 工具发现与编排 | 工具增多后，发现延迟、命名冲突、权限模型可能需要演进。 | 工具分组、版本化元数据、与 `GlobalConfig`（如默认 MCP 端口配置）一致的可配置性。 | Agent、网络 |

---

## 5. 可观测性、调试与性能（observability / performance）

| 方向 | 为什么重要 | 大致思路 | 技能领域 |
|------|------------|----------|----------|
| 统一剖析入口 | 贡献者需要低成本回答「慢在哪里」。 | 沿用并扩展 `docs/development/profiling_dimos.md` 中的实践（如对特定 blueprint 使用 py-spy），补充典型瓶颈清单。 | 性能、SRE |
| 结构化日志与运行注册 | 已有每 run 的日志与注册信息（见 CLI/AGENTS 说明）；可进一步强化关联 ID、跨进程追踪。 | 在不大改接口的前提下，统一关键生命周期事件的字段。 | 核心、CLI |

---

## 6. 测试与 CI（testing / CI）

| 方向 | 为什么重要 | 大致思路 | 技能领域 |
|------|------------|----------|----------|
| 默认套件与重型标记的平衡 | `pyproject.toml` 中 `addopts` 通过 `-m` 排除了 `tool`、`self_hosted`、`mujoco` 等标记，便于本地快速反馈；自托管与仿真测试仍有维护成本。 | 审视标记粒度、`./bin/pytest-fast` 与 `./bin/pytest-slow` 的文档是否与贡献者习惯一致；减少「只能用某一种命令才能绿」的摩擦。 | 测试、DevOps |
| 仅文档改动的 CI 语义 | `.github/workflows/ci.yml` 在 `push`/`pull_request` 上配置了 `paths-ignore: '**.md'`；该工作流里包含文档代码块的 `md-babel` job（仓库内执行 `./bin/run-doc-codeblocks --ci --no-cache`）。因此**仅有 Markdown 变更的提交/PR 可能不会触发整条 `ci` 流水线**（含 `md-babel`）。 | 文档贡献者可本地对照 `docs/agents/docs/codeblocks.md` 运行 `md-babel-py`，或在仓库根目录执行 `./bin/run-doc-codeblocks --ci`，与 CI 一致时可加 `--no-cache`。若要讨论「仅改 Markdown 时是否仍需触发整条 `ci`」，可关注 `paths-ignore` 与拆分工作流的利弊。 | DevOps |

## 7. 文档与开发者体验（docs / DX）

| 方向 | 为什么重要 | 大致思路 | 技能领域 |
|------|------------|----------|----------|
| 可执行示例与 md-babel | `docs/agents/docs/codeblocks.md` 要求文档中的代码块可执行；这对正确性有利，但也提高写作门槛。 | 提供更多带 `skip` 等标注的范例模板；对纯叙述性页面保持无「可执行块」或减少块数量。 | 文档 |
| 跨语言读者 | 本仓库既有英文主文档，也有中文 issue/讨论需求。 | 在关键 developer 文档维护简短双语标题或链接，避免整树重复翻译。 | 文档 |
